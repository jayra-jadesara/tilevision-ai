feat: Task 1 (Persistent Indexed Folder), Task 2 (Smart Re-index), and the unreadable-dropdown bug (Tasks 5/10)

TASK 1 - Persistent Indexed Folder:
Previously, restarting the app left the Index page completely empty even
though the catalog was fully intact in SQLite - the app never remembered
which folder had been indexed. Fixed by:
- New indexed_folders table (additive migration, no data loss) tracking
  folder_path + last_indexed_at.
- New IIndexedFolderRepository / SQLiteIndexedFolderRepository, following
  the existing repository pattern (IImageRepository/ILicenseRepository).
- IndexImagesUseCase optionally accepts a folder_repository and records
  the folder as indexed after each successful (non-cancelled) scan;
  get_last_indexed_folder_status() combines the persisted record with a
  live tile count for accurate display.
- IndexingViewModel.load_persisted_folder_state(), called once when
  IndexingView is constructed, restores "Folder: X / Indexed Images: N /
  Status: Ready / Last Indexed: Today 4:35 PM" - verified end-to-end with
  a simulated app restart (fresh ViewModel/View over the same DB).
- Fixed a real bug found while building this: SQLite's CURRENT_TIMESTAMP
  only has second-resolution, so recording two folders within the same
  second (trivially possible, and exactly what the first test run hit)
  tied and sorted unpredictably. Switched to strftime with fractional
  seconds for millisecond resolution.

TASK 2 - Smart Re-index:
Clicking Start Indexing on an already-indexed folder previously only
distinguished "indexed" vs "skipped" and had no concept of deletion at
all - a file removed from disk stayed in FAISS/SQLite forever. Now:
- scan_and_index_directory() returns a structured ScanResult
  (new/modified/deleted/skipped/failed counts, elapsed + time_saved
  seconds) instead of a flat 3-tuple.
- New deletion detection: after a scan, any previously-indexed tile under
  the scanned folder that wasn't seen on disk this pass is removed from
  both FAISS and SQLite. Skipped entirely on a cancelled scan (a partial
  file listing would otherwise look like mass deletion).
- New vs modified are now tracked separately (previously both just
  counted as "indexed").
- "Time Saved" is estimated from this scan's own observed per-file
  timing (falls back to a fixed heuristic when nothing was actually
  processed, i.e. everything was already up to date).
- IndexingView shows all of this: New / Modified / Deleted / Skipped
  stat cards, a "Saved ~Xs by skipping unchanged files" line, and
  "Everything is already indexed." when a re-scan finds no changes at
  all, instead of a wall of zeros.
- IndexingWorker/IndexingViewModel's indexing_finished/indexing_completed
  signals now carry the full ScanResult object rather than 3 raw ints -
  updated both call sites and all existing tests to match.

TASKS 5 & 10 - Unreadable dropdown text (root cause fixed app-wide):
QComboBox's popup list (QAbstractItemView) was never styled anywhere in
the app. It inherited the app-wide `color: #E8EAF6` (near-white) text
rule but NOT a matching background - Qt's default popup background
stayed native (frequently white), producing near-white text on a
white/light background. Fixed at the QApplication level in
theme_manager.py (both dark and light variants) rather than patching
each view's local stylesheet individually, so it's a single fix covering
every dropdown app-wide, including ones added later. Also added matching
QMenu (context menus), QTableWidget selection/hover, and QScrollBar
styling while in there, since they had the same "never actually styled"
gap. Made theme application in MainWindow unconditional (previously
skipped entirely if no Settings instance was passed in) so this fix
always applies, not just when Settings happens to be wired up.
